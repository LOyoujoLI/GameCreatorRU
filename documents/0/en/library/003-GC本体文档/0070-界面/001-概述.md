# Interface System Overview
The interface system is mainly used for the various interfaces displayed in the game.

<img src="img/t44.png" />


### Multiple pre-set basic components
Button component, image component, container component, text component, player value variable component, avatar component, animation component, input text component, checkbox component, switch component, tab component, slider component, sub interface component, list component, dropdown box component, standing avatar component, etc.<br>
>The text component, the player value variable component, and the switch component can be bound to the player's variables in order to display the values of the variables in real time.


### Component binding custom trigger events
It supports binding custom trigger event categories, such as click events, mouse hover events, etc., with a custom trigger category name and then implementing the trigger in the script. <br>Multiple events can be executed simultaneously if the execution conditions are met. <br>
>For example, there are 3 types of events in the "Feather of cyan" template: "click event", "mouse hover event", "mouseover event"

<img src="img/t45.png" />

### Interface binding script
The author-generated interface automatically generates the corresponding GUI_XXX class, which can be accessed directly to manipulate the controls. <br> 
Also if you create your own class that inherits from the system-generated GUI_XXX class, you can write more logic. <br>
> For example, the system settings screen in the "Feather of cyan" is bound to a user-defined class called GUI_Setting.<br>
> In the constructor of this class you can directly access and manipulate the tab component (typeTab), and since the system automatically generates the relevant code, code completion and code hints, etc. are also available and will be very convenient.

### System group interface
System group interfaces are interfaces that are opened by events or using scripts (GameUI.show), are recorded in system preset groups, are unique, and can be managed by the ID of the interface.<br>
>For example, screen #3 is opened by an event, and screen #3 closed by an event is the previously opened screen #3.
>You can also create a new interface instance by scripting it yourself: e.g. new GUI_XX or GameUI.load(x,true);
