# Plugin creation and distribution
Plugin creators are invited to read this document when creating plugins in order to learn how to create plugins as well as to understand the specifications.

## Plugin Release Specification
1.You can't reference the module or data or material resources that come with the template, please uncheck the reference item.<br>

2.Please make sure to create your own folder for the material resources and name it as follows: custom_{user_number}, e.g. asset/image/kds_1/xxx.png so that it will not conflict with other plugins or templates with material. <br>

3.When including [Some properties of the scene object model] or [World settings] or [Player settings], use complex names that are not easily renamed if possible, e.g. use variable name_{user_number}, the presence of renamed properties will prevent the plugin from being installed. <br>

4.For content with categories, please make it to 16 categories-insert for unified management.

6.Use complex naming for class names in scripts whenever possible, and do not easily rename variable names, such as using class name _{user number}, which may cause conflicts and lead to game errors if classes with the same name exist.<br>

7.When referencing various modules or data in the script, to ensure that the number corresponding to the user's installation can be found, please refer to [Appendix: Defining Data Numbers](#Defining Data Numbers).<br>
  > For example, if you reference interface #5, if this interface #5 is the one you want to package into the plugin together, the user will install it and insert it into an empty location, and the number will change. <br>
  If this interface 5 is not packaged into the plug-in, the system does not replace the number, but still uses the interface 5 of the user's current project.

## Publish Plugin

### Open the Publish Plugin screen
Menu-Plugin system-Publishing plugins

### Add the Manage Plugin Content element
Select Add plugin content element to add the corresponding element (only if it exists in the current project). <br> 
For manually associated ones, you can use the right mouse button to delete them, and for automatically associated ones, you can check the box to decide whether to package them or not,<br>
For example, the template comes with data and materials that need to be unchecked, refer to the [[Plugin Release Specification](#Plugin Release Specification)]. <br> 
==Please check the packaged resources to make sure that only plugin related content is included correctly when releasing. ==

### About the preview plugin effect
Checking the Preview plugin effect will upload the current project's game together as a demo effect.

### Update Plugin
If you are updating a plugin and not releasing a new one, please select the plugin to the right of the plugin name.

## Appendix
### Define the data number
Define the variable name anywhere in the script packaged into the plugin according to the specification, and call it elsewhere. <br> 
==Conforming to variable name naming rules is sufficient (must come with type and assignment, value must be a constant). ==<br> 
For example, to call interface number 5, define a static variable of class: static PLUGIN\_GUI\_custom named:number = 5,<br> 
so that the system can find and replace it with the correct number when the plugin is installed. <br> 
Here are examples of the definitions in each place:<br>
>// Static variables defined in the class<br>
static PLUGIN\_GUI\_Custom Naming:number = 5;<br><br>
// Instance properties defined in the class<br>
private PLUGIN\_GUI\_Custom Naming:number = 5;<br>
public PLUGIN\_GUI\_Custom Naming:number = 5;<br>
protected PLUGIN\_GUI\_Custom Naming:number = 5;<br><br>
// Define external global variables or internal local variables<br>
var PLUGIN\_GUI\_Custom Naming:number = 5;<br>
let PLUGIN\_GUI\_Custom Naming:number = 5;<br>

### Rewrite functions that already exist
Creating a plugin may involve the need to extend the original script. By overriding functions,<br> 
overriding the static methods of a class, suppose the myFunc method in a class named ABC.
>// Record the original function<br>
var oldAbcMyFunc = ABC.myFunc;<br>
>// Rewrite the function<br>
ABC.myFunc = function(){<br>
&nbsp;&nbsp;// Execute the old function, the scope and parameters are guaranteed to be the current call, and the order of the calls can be modified arbitrarily, or not execute the line when it means that no old logic is executed<br>
&nbsp;&nbsp;oldAbcMyFunc.apply(this,arguments);<br>
&nbsp;&nbsp;// Implementing new logic....<br>
&nbsp;&nbsp;XXXXXXXXXXXX<br>
}

Override the instance methods of the class. Suppose the class named ABC has the instance method myFunc:<br>
>// Record the original function<br>
var oldAbcMyFunc = ABC.prototype.myFunc;<br>
>// Rewrite the function<br>
ABC.prototype.myFunc = function(){<br>
&nbsp;&nbsp;// Execute the old function, the scope and parameters are guaranteed to be the current call, and the order of the calls can be modified arbitrarily, or not execute the line when it means that no old logic is executed<br>
&nbsp;&nbsp;oldAbcMyFunc.apply(this,arguments);<br>
&nbsp;&nbsp;// Implementing new logic....<br>
&nbsp;&nbsp;XXXXXXXXXXXX<br>
}
