# Code example: scene object related
Scene object includes scene object data and object entity. Scene object data is mainly used for storage reading and transmission, which generally refers to SceneObject, and also includes custom properties of scene object.<br>
Scene objects generally occupy the list of scene objects in the scene in a placeholder manner (scene.sceneObjects).<br>
Scene objects that belong to a player can find the corresponding player by so.player.<br>
Instances of scene objects are created based on their prototype binding to the scene object class.<br>

# Client-side scenario objects
#### Stand-alone version: Create scene objects
**Create an object by cloning an object in a scene:**<br>
The properties of the cloned object are identical to those of the predefined source object.<br>
>var fromSceneID = 1; // ID of the corresponding scene<br>
>var sceneObjectID = 5; // The ID of the pre-positioned scene object in the scene<br>
>var soc: ClientSceneObject = Game.currentScene.addSceneObjectFromClone(fromSceneID, sceneObjectID, true) as any;<br>

**Create the object by means of New:**<br>
New object properties need to be filled by themselves, custom properties also need to be filled by themselves, if the custom properties and display objects are many and complex, it is recommended to use the cloning method, cloning and then modify the necessary properties can be.<br>
>// New scene object data<br>
>var so = new SceneObject();<br>
>// Set the basic properties of the scene object:<br>
>so.avatarID = 1; // Avatar set to avatar #1<br>
>so.avatarAct = 1; // Play #1 action<br>
>so.modelID = 1; // Using prototype #1<br>
>// Add to scene object, create entity object from data, return value is entity scene object<br>
>var soc = Game.currentScene.addSceneObject(so,false,true);<br>
>soc.x = 500;<br>
>soc.y = 500;<br>
>soc.自定义属性 = xxx;<br>

### Stand-alone version: Remove scene objects
Default scene objects are automatically removed from the scene if they do not meet the occurrence conditions, but are not removed from the record list (scene.sceneObjects).<br><br>
**Removed from the current scene and removed from the record list:**<br>
>// so is the scene object data or entity, find the entity object at that location by so.index in order to remove<br>
>Game.currentScene.removeSceneObject(so, true):

**Removal from the current scene, but not from the list of records:**
>//  so is the scene object data or entity, find the entity object at that location by so.index in order to remove<br>
>Game.currentScene.removeSceneObject(so, false):

#### General: Play and unplay animations on scene objects
**Animation played on the scene object:**
> playAnimation(aniID: number, loop: boolean, isHit: boolean, fps: number = null, superposition: boolean = false): Animation

**Stops the animation on the object:**
> stopAnimation(aniID: any): void;



#### Web version: Create scene objects
The network version of the scene object data is on the server side, and the client scene scene object is generally created from the server after the scene object data is transferred to the client.<br>
You can refer to the GameCreator2DNet kernel project. After logging in and connecting to the server, the server will transfer relevant scene object data for creation, and you can set permissions when customizing properties in order to make, for example, which properties are public and which properties are private.

# Customizing object behavior
Object behavior is customizable and must be inherited from the SceneObjectBehaviors class to implement specific logic.<br>
For example, the GameSceneObjectBehaviors class is used in the "ARPG - Book of black" template to implement the specific behavior of the object.<br>
> class GameSceneObjectBehaviors extends SceneObjectBehaviors

Implement a logical pause state determination: in order to determine which actions are required to wait for the end before continuing. (For example, in an RPG game, if the object is moving, wait until it finishes moving before performing the next step).
>protected get logicPause(): boolean {<br>
>&nbsp;&nbsp;&nbsp;&nbsp;// to do
>}<br>

Implement the reset function: so that the behavior editor can be reset when previewing as well as jumping to steps.<br>
>reset(defSceneObejct: SceneObject): void {}

Implement specific behaviors, corresponding to the custom behavior number, #1 custom behavior then execute behavior1, #2 then execute behavior2, and so on.<br>
And the panel properties in the custom behavior automatically correspond to parameters, which are loaded in order.<br>
It is also necessary to implement an "ignore process" for the behavior editor preview（this.ignoreProcess，The reason is that when previewing in the editor, you need to skip the process and preview the result directly, such as the act of moving to a certain coordinate.）<br>
> // For example, three parameters x, y, and whether to use grid coordinates are provided in the custom behavior editor for the user to enter.<br>
